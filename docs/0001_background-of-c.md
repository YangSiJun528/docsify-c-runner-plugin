# C언어의 배경

## C언어의 탄생

- 1971-1972년에 Unix 운영체제를 PDP-11 미니컴퓨터로 포팅하는 과정에서 개발
    - Dennis Ritchie가 Ken Thompson과 함께 개발
    - Ritchie가 C 프로그래밍 언어의 원래 버전을 완성했으며, 이후 Thompson이 UNIX 커널을 C로 다시 작성
    - 이전 Unix는 어셈블리어로 만들어져 있었음
- PDP-11는 16비트 아키텍처 컴퓨터이다. (long이 32비트인 이유임. 이때는 int가 16이였으니까)
- 이게 얼마나 오래된거냐면, 우리가 잘 아는 다른 표준과 비교했을 때
    - TCP/IP는 1974년에 논문으로 쓰였고, 표준으로 확립된건 1983/1/1
    - IEEE 754는 1985에 처음 제정됨
    - 자바는 1995에 썬 마이크로시스템즈에서 발표됨
    - ARPANET(인터넷의 탄생)이 1969년이였음

## C언어의 발전

- C89/C90 표준 (ANSI C)
    - C89/C90 (ANSI C 또는 ISO C)는 1989년과 1990년에 각각 발표된 C언어의 첫 번째 표준화된 버전
    - 30년이나 지났지만 아직도 대부분의 컴파일러가 지원하는 표준
    - 오늘날 대부분의 C 코드도 이 버전
        - 많은 임베디드 시스템(C의 주 사용분야)은 여전히 89만 지원
        - 임베디드에 돌아가는 OS용 컴파일러는 대부분 C89 표준을 대상으로 구현됨
    - 문서
        - ISO와 ANSI는 표준 문서를 약 $100로 구매해야하기 떄문에 비공식 자료를 주로 참고한다.
        - 다만, 전 세계 연구자나 교수들이 교육 목적으로 공유하는 경우가 많아, 내용 자체는 진본과 거의 일치하는 경우가 많다곤 함.
        - 앞으로의 내용에서 "공식 문서에선 ~" 이나 "언어 표준에선 ~" 같은 내용이 나오면 아래 문서를 참고하였다고 보면 됨.
        - [비공식 공개 사본 ISO/IEC 9899:1990 문서](https://www.yodaiken.com/wp-content/uploads/2021/05/ansi-iso-9899-1990-1.pdf)

- C99 표준
    - 1999년에 ISO/IEC 9899:1999로 발표된 새로운 ISO C 표준 버전
    - 새로운 데이터 타입과 기능들이 추가됨
    - 주요 변경점 (실 사용 위주)
        - 변수 선언 어디서나 가능
        - 한줄 주석 `//` 추가
        - 루프 내부 변수 선언 가능
        - IEEE 754 부동소수점 지원 (강제는 아님, 다른 부동소수점 사용 가능)
        - `<stdint.h>` 도입
    - 문서
        - [비공식 공개 사본 ISO/IEC 9899:1999 문서](https://www.dii.uchile.cl/~daespino/files/Iso_C_1999_definition.pdf)
            - Universidad de Chile 도메인 사이트에서 가져옴.

- C11 표준
    - C11은 주로 당시 일반적인 컴파일러들이 이미 지원하고 있던 기능들을 표준화
    - 멀티스레딩 지원: `<threads.h>` 헤더 추가
    - 원자적 연산: `<stdatomic.h>` 헤더와 `_Atomic` 타입
    - 개선된 유니코드 지원: `char16_t`, `char32_t`
    - 익명 구조체/공용체
    - 타입 제네릭 매크로: `_Generic` 키워드

- C23 표준
    - (생략)

사이에 몇 버전이 있긴 한데, 그리 큰 변경은 아니라 생략.

## C를 배워야 하는 이유

- 최근에도 높은 점유율, C가 나오고 나서부터 Top 5안에 항상 듦
- 언매니지드 언어로써 학습/컴퓨터 동작의 이해
- 많은 언어의 참고가 됨 - C-Like 언어들
- 언어의 특징과 생태계로 인해서 대체 불가능한 언어
    - Rust는 용도가 다르고, Zig가 있지만 아직 많이 부족함.

## C언어의 문제점과 한계
(개인적인 의견 포함)

여전히 사용될 분야는 존재하지만, 높은 생산성과 편의성을 제공하는 최신 언어들이 많아 범용적으로 사용하기에는 한계가 있다.

또한 오래 된 만들어진 언어의 특성상 안전한 개발을 위한 기능이 부족하고, 실수를 유발하는 요소들이 존재함.

### 주요 문제점들:
- 수동 메모리 관리: 메모리 누수, 댕글링 포인터 등의 위험
- 엄격하지 않은 표준 정의: 구현에 따른 차이 발생
- 복잡한 매크로 시스템: 디버깅과 유지보수의 어려움
- 언어 특성으로 인한 생산성 문제: 반복적인 코드 작성 필요
- 진입 장벽: 오래된 생태계로 인한 복잡성

### UB, UsB, IdB (Undefined, Unspecified, Implementation-defined Behavior)

- Undefined Behavior (UB): 표준에서 결과를 정의하지 않은 동작으로, 어떤 일이 일어나도 유효함
    - 절대 작성하지 말 것
- Unspecified Behavior (UsB): 어떤 선택지를 따를지는 정의되지 않고, 구현마다 달라질 수 있으며 문서화되지 않아도 되는 동작
    - 절대 작성하지 말 것
- Implementation-Defined Behavior (IdB): 표준에서 결과가 구현(컴파일러/플랫폼)에 따라 정의되도록 허용하지만, 반드시 문서화해야 함
    - 사용하되 의존하지 말 것, 문서를 참고할 것

큰 프로젝트에서는 이러한 동작들에 전혀 의존하지 않는 것이 불가능에 가까워, 매크로 등을 사용해 환경에 맞게 여러 구현을 작성하고 선택적으로 빌드할 수 있게 합니다.

### 수동 메모리 관리의 위험성

수동 메모리 관리로 인한 생산성 및 유지보수성 저하는 심각한 문제이다.

특히 보안 취약점의 상당 부분이 메모리 안전성 문제에서 기인한다.

- https://news.hada.io/topic?id=13581
- https://security.googleblog.com/2024/09/eliminating-memory-safety-vulnerabilities-Android.html

### 제한적인 문법

- 접근 제어자, 함수 오버로딩, 인터페이스 등의 현대적 기능 부재
- 이러한 기능들을 우회적으로 구현하는 방법은 존재하지만, 언어 차원의 지원 부족
- 대규모 코드베이스에서의 작업을 어렵게 만드는 요인

## 관련한 사담
(여기서부턴 전부 개인적인 생각임)

취미/공부 목적으론 좋으나, 실무 용도는 별로다. 취미도 생산성을 고려하면 별로다.  
하지만 이런 자유로운 언어는 매력적이니까, 아니라고 생각하면 다른 언어를 쓰자.  

대부분의 실무에서 C로 짜야할 정도로 높은 효율이 필요한 경우는 적다. 
또한 실무에선 대규모 코드베이스에 유리한 여러 기능을 제공하는 C++을 많이 사용했고,    
요즘엔 Rust나 Go로도 많이 넘어가는 추세. MS, Google, AWS 등에서도 Rust를 많이 밀어주고 많이 사용함.   

현대 언어와 비교해 문법 슈거(syntactic sugar)도 적고, 언어 자체에서 툴링을 지원하거나 하나의 널리 쓰이는 선택지가 부족한 느낌.    
그런 툴링이 필요하지 않거나 비효율적인 환경(주로 임베디드)인 것도 한몫한다고 생각함.  

구문 슈거가 많으면 컴파일러를 구현하기 어려웠을 거고, 성능도 부하가 심했을 거임.   
요즘 맥북으로도 Rust 좀 무거운 프로젝트 돌리면 수십분 걸릴 때도 있는데, 70~80년대 컴퓨터에서 이런 무거운 컴파일러를 사용할 수 있을리 없었을 테니까.

이런 역사적인 배경이 있고, 지금도 그런 예전 환경에서 돌아가는 게 중요하다보니 기능을 추가하지 않았을거임.
